# ACPs Partner Agent 需求分析

## 1. 项目目标

基于 `ACPs-community` 规范开发并注册一个可互联的 `Partner` 智能体，首期目标是跑通最小可用闭环：

1. 实现符合 AIP 的 `Direct RPC` 交互能力；
2. 完成 ATR 可信注册并获取可用证书；
3. 具备可被平台发现和调用的基础能力描述（ACS）。

## 2. 已确认范围

### 2.1 角色与形态

- 角色：仅做 `Partner`（不做 Leader）。
- 形态：`实体 Agent`（非 ontology 本体）。

### 2.2 交互模式

- 首期只做 `Direct RPC`。
- 暂不做 `Group` / RabbitMQ。

### 2.3 业务能力边界

- 业务能力：聊天咨询。
- 输入输出：仅支持文本。
- 明确不支持：图像、语音等多模态输入输出。

### 2.4 高级能力开关

- `streaming = false`
- `notification = false`
- `messageQueue = []`

### 2.5 证书策略

- 首期只申请：`serverAuth`。
- 暂不申请：`clientAuth`（后续如需主动访问他方服务再补）。

### 2.6 部署基础信息（已确认）

- 公网 IP：`113.47.5.136`
- 服务器已对公网开放端口：`80/443/8080/5000`（另有管理端口 `22/3389`）

## 3. 技术可行性结论（重点：IP URL + 无域名）

## 3.1 结论

在当前 ACPs 开源实现下，**技术上可行**，即可以使用 `IP` 形式的 endpoint URL 完成开发与注册链路。

## 3.2 依据

- `registry-server` 的 ACS 校验核心检查是：
  - URL 结构是否合法；
  - transport 与 URL scheme 是否匹配；
  - endpoint 是否声明 `mutualTLS` 安全方案。
- 该实现并未在代码层强制“必须有域名备案才能提交 ACS”。
- `ca-server` 支持从 `ACS.certificate.altNames.ip` 读取 IP SAN 参与证书签发。

## 3.3 关键技术约束

- 若使用 `serverAuth` 且证书无 `DNS/IP SAN`，会出现 TLS 主机名校验失败风险（代码中有告警）。
- 因为你使用 IP 访问，建议在 `ACS.certificate.altNames.ip` 中显式配置服务器公网 IP。
- 首期不做 mTLS 实战联调时可先本地调通；上线联调阶段必须按平台要求启用 HTTPS/mTLS。

## 4. 首期功能性需求

### 4.1 AIP 命令支持（必须）

Partner 需正确处理以下命令并返回 `TaskResult`：

- `start`
- `get`
- `continue`
- `complete`
- `cancel`

### 4.2 状态机要求（必须）

- 合法状态覆盖：`accepted / working / awaiting-input / awaiting-completion / completed / failed / rejected / canceled`
- 终态（`completed/failed/rejected/canceled`）不可被非法改写；
- 非法状态下的命令需按协议做忽略或安全返回（保持幂等）。

### 4.3 数据对象要求（必须）

- 输出结构符合 AIP 对 `Message / TaskCommand / TaskResult / TaskStatus / Product / DataItem` 的定义；
- 文本内容统一用 `TextDataItem` 表达；
- `senderRole/senderId/taskId/sessionId` 字段语义一致。

### 4.4 ACS 要求（必须）

至少完整覆盖以下字段并与实现一致：

- `protocolVersion`
- `name/description/version`
- `provider`
- `securitySchemes`
- `endPoints`
- `capabilities`
- `defaultInputModes/defaultOutputModes`
- `skills`
- `certificate`（建议配置 `altNames.ip`）

## 5. 非功能与工程要求

- 先保证最小可运行，再逐步增强，不引入 demo 的复杂 LLM 编排。
- 首期任务状态可先内存存储；后续可升级到持久化存储。
- 必须提供可验证路径（命令与状态机测试、基本联调脚本）。

## 6. 注册与上线流程需求（ATR）

目标链路：

1. `auth login`
2. `agent save`（提交 ACS 草稿）
3. `agent submit`（发起审核）
4. `agent check/sync`（拿到并同步 AIC）
5. `cert eab fetch`
6. `cert issue --usage serverAuth`

注意：证书申请依赖审核通过后的 AIC 与 EAB。
